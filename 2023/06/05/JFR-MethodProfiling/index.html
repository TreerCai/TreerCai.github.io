JFR MethodProfiling

JFR MethodProfiling Event 简介

JMC 的 “方法分析” 页用于查看特定方法的运行频率以及运行方法所需的时间。可以帮助用户通过识别执行时间过长的方法来确定方法瓶颈。

方法分析页面的数据来源是 JFR 记录的 ExecutionSample 事件信息与 NativeMethodSample 事件信息。本文从源码角度分析 JFR 方法采样方式的独特性。如有偏颇，请指正。

JFR MethodProfiling Event 采样算法文字描述

  • 另起 JFR 线程采样线程，开始持续采样。
  • 获取当前时间
  • 判断信号量 _semaphore 是否大于 0 (初始 = 1)
    • 大于 0 _semaphore 减去 1 (获得锁可运行)
    • 等于 0 阻塞当前线程直到 _semaphore 大于0 (用于关闭采样线程)，重新获取到信号量时，更新当前时间
  • 持续采样，根据上次采样结束的时间 + 设置的间隔时间 = 下次采样的时间
  • 判断 下次采样的时间 是否大于 当前时间
    • 大于当前时间说明还不能开始采样，sleep 。睡眠时间 = 下次采样的时间 - 当前时间
    • 小于说明可以开始这一次的采样了
  • 从上次采样的线程，对往后固定个数(Java 5 Native 1)个 Java 线程进行采样
    • 获取上次采样最后的线程在 Java 线程列表中的线程号，并从下一个线程开始（如果上次遍历完列表 或 这是第一次遍历，则从列表第一个线程开始）采样，采样个数固定(Java 5 Native 1)。由于是传址调用，采样结束后最后一个被采样的线程会被记住，方便下次采样。
  • 采样结束，更新采样结束的时间，回到持续采用。

JFR MethodProfiling Event 采样算法源码分析

方法采样频次控制

ExecutionSample 与 NativeMethodSample 事件属于采样事件，需要按照一定频率进行。 Hotspot 会起一个专门的线程按照一定频次完成采样任务，一次采样任务只会采集固定个数的指定类型线程，并记录最后采集的线程；等下一次采集任务发生时会根据上次采集的线程，继续采集固定个数的指定类型线程，知道线程队列都被采集过一遍，再从头开始采集。
